    File: scripts/commands/analyze.sh:53,94,100,113,127,147

    The token is passed via curl -u "$TOKEN:" and sonar-scanner -Dsonar.login="$TOKEN".
    Both expose the credential in `ps aux` / /proc/PID/cmdline to ANY local user/process

      CREDS=$(printf '%s' "$TOKEN:" | base64)
      curl -s -H "Authorization: Basic $CREDS" "$URL"

 FINDING 1 — orchestrator.py:312
 `--yolo` auto-approves ALL tool execution for spawned agents
 SEVERITY: HIGH
─────────────────────────────────────────────

```text
cmd = [HERMES_CMD, "-z", prompt, "--yolo"]
```

The orchestrator spawns up to 3 parallel Hermes subprocesses, each with `--yolo` (skip-all-confirmations). Every spawned agent can execute arbitrary terminal commands, write files, and make network calls with zero user approval. If a task string or codebase content influences the agent's output (prompt injection from a malicious file in the repo), the agent acts on it autonomously.

This is the single highest-risk item. An agent reviewing a file containing adversarial instructions (e.g. "run this curl pipe bash") would execute it without pause.

Recommendation: Remove `--yolo` for the default path. If无人attended execution is genuinely needed, gate it behind an explicit `--i-know-what-im-doing` flag and log every command the spawned agents run.

─────────────────────────────────────────────
 FINDING 2 — analyze.sh:53,100,113,127
 SonarQube token leaks into process list
 SEVERITY: MEDIUM
─────────────────────────────────────────────

```text
curl -s -u "$TOKEN:" "$SONAR_URL/api/system/status"
```

`curl -u user:pass` passes credentials as argv, visible to any local user via `ps aux` or `/proc` during the window the curl runs. Four call sites do this.

Recommendation: Use `curl -H "Authorization: Bearer $TOKEN"` instead — credentials go in the HTTP header, not the process argument list.

 FINDING 3 — pre-commit.sh:12
 Silent `git add -u` stages unintended changes
 SEVERITY: MEDIUM
─────────────────────────────────────────────

```text
python3 scripts/fix-markdown.py . 2>/dev/null
git add -u 2>/dev/null
```

The markdown fixer runs across the entire repo, then `git add -u` re-stages ALL modified tracked files — not just the ones the user staged. If a developer had unstaged WIP changes in a tracked `.md` file, those get silently swept into the commit. The `2>/dev/null` hides any errors.

Recommendation: Only re-stage files that were already in the index:

```text
git diff --cached --name-only | grep '\.md$' | xargs -r python3 scripts/fix-markdown.py
git add -u -- '*.md'
```

Or scope the fixer to `$(git diff --cached --name-only -- '*.md')` only.

 FINDING 4 — analyze.sh:157-206
 Python code injection surface via $PROJECT_KEY
 SEVERITY: LOW (local-only, but fragile)
─────────────────────────────────────────────

The inline Python script embeds `$PROJECT_KEY` (a bash variable) directly:

```text
print('  → Check http://localhost:9000/dashboard?id=$PROJECT_KEY')
```

If PROJECT_KEY contains a single quote or backslash, the embedded Python breaks or executes injected code. PROJECT_KEY defaults to a safe literal, but it's user-controlled via `$2`. Since this is local-only, impact is limited to self-injection, but it's a correctness/robustness hazard.

Recommendation: Pass `$PROJECT_KEY` as a Python argument (`sys.argv[1]`) rather than string-interpolating it.
